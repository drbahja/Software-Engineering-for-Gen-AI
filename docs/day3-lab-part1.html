Is This AI Project Feasible? 🔍

Before a single line of code: learn to judge whether an AI project can succeed, and to define a realistic first version (MVP).

Step 1 — Understand the 4 feasibility pillars (10 min)

Companies lose millions starting AI projects that were never feasible. Professionals check four pillars before committing. For each pillar, ask yourself the guiding questions:

PillarGuiding questions
1. Technical feasibilityDo the tools/models to solve this exist? Does the team have (or can they get) the technology? Is the problem actually solvable with today's AI?
2. Organizational readinessAre there enough skilled people? Does leadership support it? Who will maintain it after launch?
3. Data availability & qualityDoes the data exist? Is it labeled? Clean? Accessible (legally and technically)? Enough of it?
4. Cost & resource constraintsIs the budget realistic for the timeline? Can existing tools be reused instead of building from scratch?

Worked mini-example: campus food delivery

A university wants AI to predict food-delivery times on campus. Quick pillar check: Technical? Yes — delivery-time prediction is a solved problem (regression). Organizational? Partially — one part-time IT person, so keep it simple. Data? Yes — 2 years of order logs with timestamps. Cost? Yes — a simple model on existing servers costs almost nothing. Verdict: feasible, if scoped small. Notice the verdict is per-pillar, with reasons — that is exactly what you'll produce below.

Step 2 — Read the real case (5 min)

Background — UnitedHealth Group. One of the largest health insurers in North America, processing over 10 million claims yearly, with rising concerns about fraud, especially in dental and outpatient claims. Leadership wants an AI model to automatically flag potentially fraudulent claims for human review — reducing unnecessary payouts and speeding up processing.

Key facts (highlight these!):

  • Data science team is small: 2 data analysts + 1 software engineer
  • Budget for one external AI consultant during the MVP phase
  • Claims data already in a cloud database (BigQuery); company uses Google Cloud tools (AutoML, Vertex AI)
  • Historical data has structured fields: procedure type, date of service, claimed amount, provider ID, region
  • Target: prototype in 6 months, starting with the most frequent, least complex claims

💡 How to read a case like an engineerEvery sentence in a case description is a clue. "Small team" → organizational constraint. "BigQuery + AutoML" → technical head start. "Structured fields" → easier modeling. "6 months" → scope must be narrow.

Task B — Define the MVP scope

An MVP (Minimum Viable Product) does the smallest useful thing well. Decide what version 1 should include and what must wait. Think about these 5 dimensions — one row each:

  • Claim types — which claims first? (hint: the case tells you)
  • Data types — structured fields only, or scanned documents too?
  • Autonomy — does the AI decide, or does a human decide?
  • Tooling — build a custom model, or use AutoML?
  • Frequency — real-time scoring, or weekly batches?
