6requirements.txt — the shopping list

Open requirements.txt. It lists every external library the project needs:

langchain-deepseek>=0.1
langchain-core>=0.3
python-dotenv>=1.0
streamlit>=1.40

🤔 Why is this file best practice?Imagine you email your code to a friend and it crashes on their laptop: "module not found". Which libraries? Which versions? requirements.txt answers that question forever. Anyone — a teammate, a server, or you in 6 months — can rebuild your exact environment with one command. This is called reproducibility, and it is one of the most important habits in software engineering.

Now install the shopping list — but first, create a virtual environment, a private box of libraries just for this project so it never conflicts with your other projects:

# macOS / Linux
python3 -m venv .venv
source .venv/bin/activate

# Windows
python -m venv .venv
.venv\Scripts\activate

You will see (.venv) appear in your terminal. Now install:

pip install -r requirements.txt

💡 Slow download?Use the Tsinghua mirror: pip install -r requirements.txt -i https://pypi.tuna.tsinghua.edu.cn/simple

7.env — the secret keeper 🔑

Our agent talks to DeepSeek, a powerful Chinese LLM. To use it, the code needs an API key — like a password that lets your program call DeepSeek's servers.

  1. In VS Code, copy .env.example and rename the copy to exactly .env
  2. 🙋 Now raise your hand — Dr. Bahja will give you the API key during the lecture.
  3. Paste it in the file, so it looks like: DEEPSEEK_API_KEY=sk-xxxxxxxx

🤔 Why a separate file for the key?Because code gets shared — pushed to GitHub, emailed, shown on slides. Secrets must never travel with the code. The .env file stays on your machine only, and the code reads the key from it at runtime.

Open the .gitignore file (in the repository root). The first lines say:

# NEVER commit your API key
.env

This tells Git: "pretend this file does not exist." Even if you push your code to GitHub, .env — and your key — stay private.

⚠️ True story, every yearStudents (and professional engineers!) accidentally push API keys to public GitHub. Bots scan GitHub for keys 24/7 and steal them within minutes. Best-practice checklist: 1) keys live in .env, 2) .env is in .gitignore, 3) share a .env.example template instead, 4) if a key ever leaks — revoke it immediately.

8Meet LangChain 🦜🔗

You could talk to DeepSeek with raw web requests — but then your code would be glued to one provider. LangChain is a Python library that gives you one common language for working with any LLM.

Three ideas are enough for today:

LangChain conceptThink of it as…
SystemMessageThe job description you give the AI: "You are a helpful assistant named Rob."
HumanMessageWhat the user says: "Who was the first computer scientist?"
model.invoke(messages)Press send — the messages go to DeepSeek, the answer comes back.

💡 Why LangChain matters (and why our code works in China)This project originally used OpenAI — which is hard to reach from China. Because LangChain gives every provider the same interface, switching to DeepSeek meant changing one import and one class: ChatOpenAI became ChatDeepSeek. The messages, the agent, the loop — untouched. That is the power of good abstractions.

9Read the agent — 4 sections 📖

Open first_agent.py. The whole program is made of four sections. Understand these four, and you understand every LLM application you will ever build.

Section 1 — The import statements

import os

from dotenv import load_dotenv                  # reads the .env file
from langchain_deepseek import ChatDeepSeek     # DeepSeek chat client
from langchain_core.messages import HumanMessage, SystemMessage

Imports are the tools we borrow before we start working:

  • os — talk to the operating system (we use it to read environment variables)
  • load_dotenv — loads your secrets from the .env file
  • ChatDeepSeek — LangChain's client for the DeepSeek model
  • HumanMessage, SystemMessage — the two kinds of messages we send to the model

💡 NoticeEvery import maps to a line in requirements.txt. That is the connection: the file promises the tools, the imports use them.

Section 2 — The DeepSeek configuration

load_dotenv()                                   # read the .env file
deepseek_key = os.getenv("DEEPSEEK_API_KEY")    # take the key out of it

model = ChatDeepSeek(
    api_key=deepseek_key,   # your secret, from .env
    model="deepseek-chat",  # which brain to use
    temperature=0.7,        # creativity: 0 = strict, 1.3 = wild
)

This is where we configure our connection to the AI. Three decisions live here:

  • Which key — loaded from .env, never written in the code
  • Which model — deepseek-chat for general chat; try deepseek-reasoner later for hard logic problems
  • How creative — temperature: at 0 the model gives the same careful answer every time; at 1.3 it takes risks. Pick based on the job: maths tutor → low, story writer → high.

Section 3 — The system and user prompt

messages = [
    SystemMessage(content="You are a helpful assistant who is extremely "
                          "competent as a Computer Scientist! Your name is Rob."),
    HumanMessage(content="Who was the very first computer scientist?"),
]

res = model.invoke(messages)   # send both → get the answer
print(res.content)

Two different voices talk to the model:

  • The system prompt is the job description. The user never sees it, but it shapes every answer: personality, rules, language, name. Change it and you change the whole application.
  • The user prompt is the actual question, different every time.

model.invoke(messages) sends both together, and the answer comes back in res.content. The rest of the file — first_agent() and run_agent() — simply wraps this one call in a loop so you can keep chatting until you type exit.

🤔 Is this really an "agent"?Today it is the simplest possible one: a model wrapped in a loop. A real agent adds memory, tools, and decisions — exactly what we build over the next three days. Every big system starts this small.

Section 4 — The UI (user interface)

Right now our agent lives in the terminal — fine for developers, scary for users. There is a second file in the project, streamlit_app.py, which gives the same agent a real web page with a text box and a button.

🏆 Exercise — use the code as your template

You discovered the agent's weakness: it has no memory. Every message is a brand-new conversation. Your job is to fix that, using first_agent.py as your template.

Part A — Give it memory (everyone)

  1. In run_agent(), create a history list before the loop, starting with one SystemMessage — invent your own persona: a Xi'an tour guide, a strict maths teacher, a movie critic…
  2. Each turn: append the user's HumanMessage to history, and send the whole list to the model.
  3. Append the model's reply to history too. (Hint: from langchain_core.messages import AIMessage)
  4. Test: tell it your name — then ask for it. If it remembers, you win.

Part B — Make it interesting (pick one)

  • 🌏 A translator that always answers in both English and Chinese
  • 🌡️ Let the user change temperature and show how answers differ at 0 vs 1.3
  • 🧠 Swap the model to deepseek-reasoner and compare it with deepseek-chat on a logic puzzle

💡 Finished early?Give the Streamlit version memory too, using st.session_state — and show the instructor.

Be ready to demo your agent in 2 minutes and explain every line you changed.
